# 第二篇｜Context 工程：让 Claude 真正「懂」你的项目

> **系列导读**：上一篇你装好了 Claude Code，跑起了第一个任务。但你可能已经发现了一个问题——第二天打开新会话，Claude 对你的项目又像第一次见面，什么都要重新解释。这一篇解决这个问题，而且解决得很彻底。

---

## 先说一个让人沮丧的现象

你用 Claude Code 做了一个任务，告诉它：

- 这个项目用的是 PostgreSQL，不是 MySQL
- 我们的 API 路由全部用下划线命名，不用驼峰
- 测试框架是 Vitest，不是 Jest
- 永远不要直接提交到 main 分支

Claude 全都记住了，做得很好。

第二天，你打开新会话。Claude 又开始给你写 Jest 测试，又建议你用 `getUserById` 这种驼峰命名，又在某个操作后问你"要不要直接提交到 main？"

**这不是 Claude 变笨了**。这是因为大语言模型本质上是无状态的——每个新会话都从零开始，没有任何记忆。你昨天告诉它的一切，今天全消失了。

解决这个问题的工具叫 **CLAUDE.md**。

---

## 一、CLAUDE.md 是什么

CLAUDE.md 是一个 Markdown 文件，Claude Code 在**每次启动新会话时都会自动读取它**，把里面的内容作为上下文加载进来。

它的作用，相当于给一个新入职的工程师写了一份项目备忘录——你不需要每次开会都重新介绍项目背景，因为他来之前就读过了。

但 CLAUDE.md 不只是"README 的 AI 版"。这是大多数人误解的地方。

Claude 读你的代码文件，可以自己推断出你用的是 React、你的目录结构是什么、你的函数命名大致风格是什么。它**不需要**你在 CLAUDE.md 里再写一遍这些。

**CLAUDE.md 真正有价值的内容，是 Claude 读代码推断不出来的东西**：

- 你的团队为什么做了某个架构决策（而不是别的选择）
- 哪些反模式是明确禁止的，以及原因
- 测试的运行命令、CI 的触发方式
- 提交规范、分支命名规则
- 项目中存在的已知技术债，不要动它
- 你个人的偏好（比如"宁可代码稍微啰嗦，也不要过度抽象"）

把 CLAUDE.md 理解为一份**行为契约**，而不是项目文档。每一行都应该改变 Claude 的行为方式——如果某句话不会改变它做事的方式，就可以删掉。

## 二、三层配置体系：分清楚什么放哪里

Claude Code 的 CLAUDE.md 有一套多层级体系，理解这个层级是用好它的关键。

### 第一层：全局配置（个人偏好）

**位置**：`~/.claude/CLAUDE.md`

这个文件里的规则**适用于你在这台机器上打开的所有项目**，不会提交到任何代码库，完全是你个人的设置。

适合放什么：

```markdown
# 我的全局偏好

## 代码风格
- 使用 2 空格缩进
- 优先使用函数式写法，避免不必要的类
- 注释用中文，代码用英文命名

## 工作习惯
- 每次修改超过 3 个文件时，先输出修改计划等我确认
- 提交信息使用 Conventional Commits 格式：feat/fix/docs/refactor
- 不要自动推送到远程，本地提交后告诉我

## Git 规范
- 永远不提交到 main 或 master 分支
- 分支名格式：feature/xxx 或 fix/xxx
```

这些是你无论在哪个项目里都想遵守的习惯，放在全局配置里，一次设置，永久生效。

### 第二层：项目配置（团队共享）

**位置**：项目根目录的 `CLAUDE.md`，或 `.claude/CLAUDE.md`

这个文件**提交到代码库**，团队所有成员共享。每次启动会话时自动加载。

适合放什么：技术栈、架构决策、团队规范、命令、禁止事项。（模板见后文。）

### 第三层：本地覆盖（个人私密）

**位置**：`CLAUDE.local.md`（在 `.gitignore` 里）

这个文件只在你自己的机器上生效，不提交给团队，用于放一些你不想共享的个人设置：

```markdown
# 我的本地偏好（不共享）

## 本地环境
- 本地数据库连接：localhost:5433（不是默认的 5432）
- 我用 pnpm，不用 npm（团队其他人用 npm）
- 我的测试环境变量文件在 .env.test.local

## 个人习惯
- 生成代码前先用中文给我解释思路，我确认后再写
```

**三层叠加**：Claude 会把三层配置都加载进来，规则叠加生效。更具体的规则（本地 > 项目 > 全局）优先级更高。

## 三、进阶：`.claude/rules/` 目录，让配置模块化

随着项目复杂度增加，把所有规则都塞进一个 CLAUDE.md 文件会让它变得臃肿难维护。Claude Code 提供了一个更优雅的解法：`.claude/rules/` 目录。

把规则按主题拆分成多个文件：

```
your-project/
├── CLAUDE.md              ← 概述 + 引用其他规则文件
└── .claude/
    └── rules/
        ├── code-style.md  ← 代码风格规范
        ├── testing.md     ← 测试策略和命令
        ├── git.md         ← Git 工作流规范
        ├── security.md    ← 安全禁止事项
        └── api/
            └── design.md  ← API 设计规范（子目录也支持）
```

然后在主 `CLAUDE.md` 里用 `@import` 语法引用它们：

```markdown
# ShopPlatform 项目

## 项目概述
一个 B2B 电商平台，使用 Next.js 14 + TypeScript + PostgreSQL。

## 规范文件
详细规则请参阅以下文件（Claude 会自动展开这些引用）：

@.claude/rules/code-style.md
@.claude/rules/testing.md
@.claude/rules/git.md
@.claude/rules/security.md
```

**这种模式的好处**：
- 不同团队成员可以维护自己负责领域的规则文件
- CLAUDE.md 保持简洁，规则文件各司其职
- 修改某类规则不会影响其他内容，PR review 也更清晰

## 四、一份经过验证的 CLAUDE.md 模板

下面是一份可以直接用的模板，覆盖了最关键的几个部分。复制这个结构，根据你的项目填充内容。

```markdown
# [项目名称] — Claude Code 配置

## 项目概述
<!-- 一两句话说清楚这是什么项目，服务谁 -->
一个面向中小企业的财务管理 SaaS，核心功能是发票管理和报表生成。

## 技术栈
- 前端：Next.js 14 (App Router)，TypeScript 5，Tailwind CSS
- 后端：Node.js 20，Express，Prisma ORM
- 数据库：PostgreSQL 16（不是 MySQL，不要生成 MySQL 语法）
- 测试：Vitest + Testing Library（不是 Jest）
- 部署：Vercel（前端）+ Railway（后端）

## 目录结构说明
- `src/app/` — Next.js App Router 页面
- `src/components/` — 可复用 UI 组件
- `src/lib/` — 工具函数和第三方集成
- `src/server/` — 后端 API 路由和业务逻辑
- `prisma/` — 数据库 schema 和迁移文件

## 常用命令
```bash
pnpm dev          # 启动开发服务器（前端+后端）
pnpm test         # 运行所有测试
pnpm test:watch   # 监听模式运行测试
pnpm db:migrate   # 运行数据库迁移
pnpm build        # 构建生产版本
pnpm lint         # 检查代码规范
```

## 编码规范

### 命名约定
- 文件名：kebab-case（如 `invoice-list.tsx`）
- 组件名：PascalCase（如 `InvoiceList`）
- 函数和变量：camelCase
- 数据库字段：snake_case（由 Prisma 自动映射）
- API 路由：REST 风格，用连字符分隔（如 `/api/invoice-items`）

### 代码风格
- 单引号，不用双引号
- 不写分号（prettier 会处理）
- 优先用 `async/await`，避免 `.then().catch()` 链式写法
- 组件优先用函数式，不用类组件
- `any` 类型是禁忌，遇到难以推断的类型用 `unknown` 然后做类型收窄

### 错误处理
- 所有 API 路由必须有 try-catch，返回统一格式：`{ success, data, error }`
- 前端用 React Query 处理异步状态，不要手写 loading/error state
- 不要把敏感错误信息直接返回给前端

## 测试要求
- 新增业务逻辑必须附带单元测试，覆盖正常路径和边界条件
- API 路由用集成测试，不要用 mock（用测试数据库）
- 组件测试只覆盖关键交互，不测实现细节
- 不要为了覆盖率而写没意义的测试

## Git 工作流
- **严禁直接提交到 main 或 develop 分支**
- 分支命名：`feat/xxx`、`fix/xxx`、`refactor/xxx`、`docs/xxx`
- 提交信息格式：`feat: 添加发票导出功能` （中文描述）
- 每个 PR 必须关联 issue 编号，格式：`Closes #123`

## 明确禁止事项（IMPORTANT）
- **不要**在没有数据库迁移的情况下修改 Prisma schema
- **不要**在前端代码里直接访问数据库
- **不要**硬编码任何 API key、密码、连接字符串
- **不要**修改 `prisma/migrations/` 目录下的已有迁移文件
- **不要**在 `src/app/` 里写业务逻辑，API 调用统一放 `src/server/`

## 已知技术债（不要动）
- `src/lib/legacy-report.ts` 是旧报表引擎，逻辑复杂，正在重写，不要修改
- `src/components/OldDatePicker/` 使用了废弃的 API，等待替换，不要扩展功能

## 工作方式偏好
- 修改超过 3 个文件时，先列出计划并等待我确认
- 遇到我没有明确说明的架构决策，先问再做，不要自行假设
- 提供多个方案时，说明每种方案的权衡，不要直接帮我做选择
```

**注意"禁止事项"部分使用了 IMPORTANT 标记**。研究表明，加上这类强调词能显著提高 Claude 遵守规则的概率——它不会被当作普通建议，而是被当作约束条件处理。

## 五、Auto Memory：Claude 自己给自己写备忘录

从 Claude Code v2.1.59 开始，有一个很少被提及但非常实用的功能：**Auto Memory（自动记忆）**。

它的工作原理是：Claude 在工作过程中，会主动把它认为对未来有价值的信息写入记忆文件——构建命令、调试过程中发现的问题、代码风格偏好、架构洞察等。**下次会话开始时，这些记忆会自动加载回来。**

你不需要手动维护，Claude 自己决定什么值得记住。

用 `/memory` 命令可以查看和管理当前的记忆内容：

```
/memory
```

Auto Memory 的意义在于：即使你的 CLAUDE.md 没有覆盖到某个细节，Claude 也能通过自己积累的经验，在后续会话里表现得越来越"了解"你的项目。

## 六、Explore → Plan → Execute → Commit：和 CLAUDE.md 配合的工作流

CLAUDE.md 解决的是"Claude 懂项目背景"的问题，但懂背景和高质量执行之间还有距离。这一段讲 Anthropic 官方推荐的四步法，重点看 CLAUDE.md 如何在每个阶段发挥作用。

### Explore（探索）

不要一上来就说"帮我做 XX"。先让 Claude 读相关代码，建立准确的上下文：

> "在我们开始之前，先读一下 `src/server/invoices/` 目录和相关的 Prisma schema，告诉我你对现有发票处理逻辑的理解，以及你认为哪些地方可能和我要做的任务有关联。"

跳过探索直接执行，是复杂任务返工的最主要原因。Claude 对代码库的理解准确度，和最终代码质量直接相关。

### Plan（计划）

切到 Plan 模式（上篇学过的 Shift+Tab），给出具体任务：

> "我需要给发票添加批量导出功能，支持 PDF 和 CSV 两种格式，导出结果通过邮件发送给用户。先给我一份实现方案，说明你会修改哪些文件、新建哪些文件，以及大致的实现思路。先不要动代码。"

**在这一步做修正，成本最低**。方向不对时，一句话就能纠正："不对，邮件发送不用自己写，我们有 SendGrid 的封装在 `src/lib/email.ts`，用那个。" Claude 更新计划，零文件改动。

CLAUDE.md 的价值在这里开始显现——它会自动遵守你的命名规范、错误处理格式、模块边界，不需要你在计划阶段反复提醒。

### Execute（执行）

计划确认后切到 Auto 模式：

> "计划看起来对，开始执行，先从后端 API 部分开始。"

执行阶段 CLAUDE.md 里的"禁止事项"和"工作方式偏好"真正起作用。比如你的 CLAUDE.md 写了"不要在 `src/app/` 里写业务逻辑"，Claude 在生成代码时会自动把逻辑放到 `src/server/`，而不是写完后等你纠正。

### Commit（提交）

执行完成后，让 Claude 帮你收尾：

> "实现完成了，帮我检查一下所有改动是否符合我们的规范，然后按功能拆分成合理的提交，写好提交信息，提交到当前分支。不要推送到远程。"

四步走下来，一个完整的功能开发从探索到提交，流程清晰，风险可控。

## 七、从 `/init` 骨架到生产级配置

上一篇介绍了 `/init`，它能自动生成一份初版 CLAUDE.md，帮你快速起步。但这一篇的核心观点是：**`/init` 生成的只是骨架**——它能识别你的技术栈和目录结构，但它不知道你的架构决策、禁止事项和团队规范。

真正让 CLAUDE.md 发挥价值的，是生成之后的手动补充。重点补三类内容：

1. **禁止事项**：哪些事绝对不要做（如"不要直接提交到 main"）
2. **隐性规则**：代码推断不出来的约定（如"API 路由用下划线命名"）
3. **工作方式**：你期望的协作模式（如"修改超 3 个文件时先列计划"）

### CLAUDE.md 是活文档，不是一次性文件

最好用 CLAUDE.md 的团队有一个共同特点：**把它当活文档维护**。

每次 Claude 犯了一个本可以避免的错误——用了 MySQL 语法、忘记了某个命名规范、动了不该动的文件——修复完之后，立刻把对应的规则加进 CLAUDE.md。

这种"每次犯错就更新规则"的习惯，会让你的 CLAUDE.md 越来越精准，Claude 的表现越来越稳定。从第一周的"每次都要纠正"，到两个月后的"几乎不用纠正"，这个进步是真实可期的。

## 八、常见的 CLAUDE.md 写法陷阱

### 陷阱一：写成 README，而不是行为规范

❌ 错误写法：
```markdown
## 项目简介
本项目是一个用于管理企业财务流程的 SaaS 平台，
使用现代化的技术栈构建，支持多租户架构...
```

✅ 正确写法：
```markdown
## 数据库
使用 PostgreSQL 16。所有查询通过 Prisma ORM，
不要写裸 SQL（除非 Prisma 不支持的复杂查询，需先问我）。
```

区别：前者描述项目是什么，Claude 自己读代码就能知道；后者告诉 Claude 应该怎么做决策。

### 陷阱二：用 @ 引用大文件

```markdown
# 错误做法——别这么写
请参考 @docs/api-complete-reference.md 了解所有 API。
```

这会把整个文档文件塞进每次会话的上下文，浪费大量 token，而且大部分内容可能和当前任务无关。

正确做法是**提示性引用**，而不是内嵌：

```markdown
# 正确做法
遇到 Stripe 相关问题，请先阅读 docs/stripe-integration.md。
```

只在 Claude 真正需要的时候，它才会去读那个文件。

### 陷阱三：规则太多，全部失效

有人认为 CLAUDE.md 越详细越好，写了几百行。结果 Claude 的注意力被稀释，反而很多规则都没被遵守。

实践检验下来，**200 行左右是一个比较合理的上限**。超过这个长度，用 `.claude/rules/` 目录拆分，或者删掉那些"说了等于没说"的内容。

### 陷阱四：写了格式规则，却没有配置 linter

```markdown
# 这样写没有意义（如果你有 Prettier 的话）
- 使用 2 空格缩进
- 行末不要有空格
- 字符串用单引号
```

如果你的项目已经配置了 Prettier 或 ESLint，这些规则会被工具自动强制执行，Claude 不需要关心它们。把 linter 能管的事交给 linter，CLAUDE.md 只写工具管不了的事。

## 本篇小结

CLAUDE.md 是 Claude Code 里最高杠杆比的配置——30 分钟写好一份，往后每次会话都能受益，它的价值随着使用时间增长复利积累。

这一篇你掌握了：

- CLAUDE.md 是行为契约，不是项目文档，每一行都应该改变 Claude 的行为
- 三层配置体系：全局（个人偏好）→ 项目（团队共享）→ 本地（个人私密）
- `.claude/rules/` 目录让规则模块化，便于团队协作维护
- 一份可直接使用的 CLAUDE.md 模板，核心是"禁止事项"和"工作方式偏好"
- Auto Memory 让 Claude 自己积累经验，跨会话持续改善
- Explore → Plan → Execute → Commit 四步工作流，复杂任务的标准流程
- 四个常见陷阱：写成文档而非规范、引用大文件、规则太多、写 linter 能管的事
